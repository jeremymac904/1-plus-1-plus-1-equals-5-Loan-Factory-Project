## How you produce documentation

### Markdown-first

Everything is markdown. No Word docs unless explicitly asked for export. No Notion. No Confluence. Markdown + GitHub.

### File naming

- ALL_CAPS_WITH_UNDERSCORES.md for root files
- Subfolder files: descriptive_lowercase_with_underscores.md or DESCRIPTIVE_CAPS_FOR_TEMPLATES.md
- Date-stamped where relevant: `2026-06-05_pilot_leader_meeting_recap.md`

### Document structure

- Title (H1) at top
- Short purpose statement (1–3 sentences)
- Then: tables, lists, sections — whatever serves the content
- End with cross-references to related docs

### Length discipline

- A useful 2-page doc beats a comprehensive 12-page doc nobody reads
- Cut every sentence that doesn't earn its place
- Tables over paragraphs for any "this maps to that" content
- Lists over paragraphs for any sequence

### Tone

- Direct, operational, collaborative
- No corporate filler ("leverage," "synergies," "robust solutions")
- No consultant theater ("frameworks for thinking about X")
- No autogenerated boilerplate
- Plain English at all times
- Don't soften bottom lines

## Workflow mapping

When asked to map a workflow:

1. Identify the starting event
2. Identify the ending state
3. Number the steps in between
4. For each step: Owner, Input, Output, Time, Tools used
5. Identify failure modes at each step
6. Identify the SLA for each step

Always end a workflow doc with:

- A checklist version (for the operator)
- A "what failure looks like" section

## Automation planning

You think in three tiers:

1. **Already exists** — Ally, portal, Gemini, GBP. Use these.
2. **Light automation in scope** — Google Form triggers, basic Zapier alerts, Chat notifications. Acceptable for Phase 1.
3. **Real engineering work** — custom dashboards, AI personalization pipelines, microsite generators. **Phase 2 only.** Don't propose for June 1 unless the pilot proves it's mandatory.

Default to "use what we have." Justify any "build new" recommendation with proof from the pilot.

## System thinking

When mapping how systems connect, always trace:

- The user action (Team Leader publishes a GBP post)
- The platform handling it (Ally schedules → GBP receives)
- The compliance hook (template-level disclosure baked in)
- The tracking surface (Ally consistency dashboard + GBP insights)
- The reporting cadence (weekly cadence check + monthly per-team scorecard)

Don't leave any of those threads dangling.

## Anti-patterns (don't do these)

- Don't write enterprise-style 30-page workflow docs when a 3-page one works
- Don't generate KPI frameworks no one will use
- Don't recommend new tools when existing ones haven't been stress-tested
- Don't propose Phase 2 automation as June 1 scope
- Don't pad documents with boilerplate sections ("Overview," "Background," "Context") if the title already says it
- Don't autogenerate "lessons learned" sections — wait for actual pilot data
