## Branching and commits

### Default: commit directly to `main`

For a 3-pod operational repo with limited collaborators, branches add overhead without much benefit. Default to direct commits to `main`.

### When to use a branch

- If you're staging a multi-file change that depends on review
- If you're making changes that could affect multiple teams' active workflows
- If you want Jeremy to review before it lands

### Branch naming

`tera/short-description` or `marketing/short-description` or `jeremy/short-description`. Lowercase, hyphen-separated.

### Commit messages

Imperative, short:

- `Add Vietnamese division template v1`
- `Update onboarding checklist for second wave`
- `Move superseded Q2 doc to ARCHIVE`

Don't write essays in commit messages.

## Documentation standards

### Length

- A useful 2-page doc beats a comprehensive 12-page doc
- Cut every sentence that doesn't earn its place
- Tables and lists over paragraphs

### Tone

- Direct, operational, collaborative
- No corporate filler
- No autogenerated boilerplate
- Plain English

### Cross-references

- Every doc that references another should hyperlink the file path
- Don't duplicate content across files — reference the source

### Naming

- Root files: `ALL_CAPS_WITH_UNDERSCORES.md`
- Subfolder files: descriptive lowercase OR `TEMPLATE_NAME.md` for templates
- Date-stamped where relevant: `2026-06-05_pilot_leader_meeting_recap.md`

## Operational communication expectations

| Channel | Use |
| --- | --- |
| This repo | Decisions, workflows, templates, scorecards, recaps |
| Google Chat (pilot rooms) | Day-to-day operations, quick questions |
| Email (formal) | Asset deliveries, leadership updates, intake confirmations |
| Pilot Team Leader meeting | Friction, wins, decisions |
| Direct DM to Jeremy | Escalations, urgent platform issues |

If you find yourself routing something through three channels, pick one. The repo is for things that need to be searchable later. Chat is for things that need a response today.

## Anti-patterns (don't do these)

- Don't create new top-level folders without talking to Jeremy first
- Don't delete files — move them to `ARCHIVE/` with a commit note
- Don't add brand voice or program strategy decisions to subfolder docs — those go in `PROJECT_DECISIONS_AND_CONTEXT.md`
- Don't write enterprise-style 30-page workflow docs when a 3-page one works
- Don't put PII (NMLS numbers, personal contact info) in committed files
- Don't put secrets, API keys, or credentials in this repo ever

## How to ask for help

Two-step process:

1. Check `PROJECT_SOURCE_INDEX.md` — chances are there's a doc for what you need
2. If not, ping Jeremy with a single sentence: "Looking for X. Doesn't exist. OK if I create [folder]/[filename].md?"

He'll reply yes within a day. Don't wait.
